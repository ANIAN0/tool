# 沙盒服务与对接说明

> **文档目的：** 作为仓库内 **沙盒部署与 Next 对接的唯一总述**（替代原 `docs/功能开发/沙盒服务部署实施文档.md`）。
> **系统概述：** 服务端为基于 **nsjail** 的 Python HTTP 服务（`sandbox-service/`），通过 **Gateway URL** 暴露会话级 **exec / read / write / heartbeat / status** API；Next 侧 `lib/infra/sandbox/session-manager.ts` 使用 **`Authorization: Bearer`** + API Key 调用。Agent Chat 在 **`SANDBOX_ENABLED=true`** 且配置合法时，由 `lib/agent-chat/runtime.ts` 直接调用 `getSandboxToolsWithContext` 返回工具集并注入 `ToolLoopAgent`。Skill Loader 在对话开始时将 Skill 文件加载到沙盒工作区。
> **技术栈：** Python 3.9+、nsjail、systemd（可选）、Next.js、`ai` SDK `tool()`、gray-matter（YAML 解析）

## 1. 系统架构

```
浏览器 → Next (Agent Chat) → lib/agent-chat/runtime.ts → ToolLoopAgent
                                                      ↓ 工具调用
                                          lib/infra/sandbox/* → HTTPS Gateway
                                                      ↓
                                          sandbox-service (Python + nsjail)
                                                      ↓
                                          每用户隔离工作区 / 单次 exec
                                                      ↓
                                          Skill 文件加载（skills/ 目录）
```

- **会话 id：** Next 侧使用 **`conversationId`** 作为沙盒会话标识传入 Gateway 路径（与 `SandboxSessionManager.exec` 等一致）。
- **用户 id：** 请求体中带 **`userId`**，供服务端做工作区隔离与配额（与实现保持一致）。
- **Skill 加载：** 对话开始时自动将 Agent 配置的 Skill 文件下载到沙盒 `skills/{skillId}/` 目录。

### 2.1 utils 模块（logger.py）

服务端日志工具模块，提供结构化日志记录能力：

| 函数/变量 | 说明 |
|------|------|
| `get_logger(name, log_file)` | 获取配置好的日志器，支持文件和控制台双输出 |
| `default_logger` | **默认日志器实例**（名称为 `sandbox`，导出供其他模块直接使用） |
| `log_request(logger, method, path, **kwargs)` | 记录请求日志（带参数格式化） |
| `log_error(logger, error, context)` | 记录错误日志（含堆栈追踪和上下文） |

**日志配置：**
- 日志目录：`/var/log/sandbox`
- 日志格式：`%(asctime)s | %(levelname)-8s | %(name)s | %(message)s`
- 文件处理器：DEBUG 级别，UTF-8 编码
- 控制台处理器：INFO 级别

**使用示例：**
```python
from utils.logger import get_logger, log_request, log_error, default_logger

# 使用默认日志器
default_logger.info("服务启动")

# 或创建自定义日志器
logger = get_logger("sandbox")
log_request(logger, "POST", "/api/v1/sessions/123/exec", userId="user1")
log_error(logger, exception, {"sessionId": "123"})
```

## 4. Gateway HTTP 契约

以下路径相对于 **`SANDBOX_GATEWAY_URL`**，与 `SandboxSessionManager` **保持一致**：

| 方法 | 径 | 用途 |
|------|------|------|
| POST | `/api/v1/sessions/:sessionId/exec` | 执行命令；body：`userId`、`code`、`language` |
| POST | `/api/v1/sessions/:sessionId/read` | 读文件；body：`userId`、`path`（相对工作区） |
| POST | `/api/v1/sessions/:sessionId/write` | 写文件；body：`userId`、`path`、`content` |
| POST | `/api/v1/sessions/:sessionId/heartbeat` | 保活 |
| GET | `/api/v1/sessions/:sessionId/status` | 状态；响应含 `status`、`lastActivity` |

**认证：** 所有请求头 **`Authorization: Bearer <SANDBOX_API_KEY>`**（见 `session-manager.ts`）。

**错误：** 非 2xx 时管理器尝试解析 JSON `error` 字段抛出异常。

### 5.1 接口抽象（interface.ts）

**功能：** 定义沙盒核心接口，支持多种实现和 mock 测试。

**核心接口与类型：**

| 接口/类型 | 说明 |
|------|------|
| `SandboxSessionInterface` | 沙盒会话接口，定义 `exec`、`readFile`、`writeFile`、`heartbeat`、`getStatus` 五个核心方法 |
| `SandboxToolProviderInterface` | 工具提供者接口，定义 `getToolsWithContext(context)` 和 `isAvailable()` 两个方法 |
| `SandboxToolContext` | 工具执行上下文（`conversationId`、`userId`） |
| `SandboxCreationConfig` | 沙盒实例创建配置（`config: SandboxConfig`、`enabled: boolean`） |
| `SandboxInstance` | 完整沙盒实例（`session` + `toolProvider`） |
| `SandboxSessionFactory` | 会话工厂函数类型 |
| `SandboxToolProviderFactory` | 工具提供者工厂函数类型 |
| `SandboxInstanceFactory` | 实例工厂函数类型 |

**设计意图：** 通过接口抽象实现**依赖倒置**，上层代码（如 `lib/agent-chat/runtime.ts`）依赖接口而非具体实现，便于后续替换沙盒后端（如 Docker、WebSocket 等）和单元测试 mock。

### 5.2 工厂函数（factory.ts）

**功能：** 提供统一的工厂接口创建沙盒实例和工具提供者，支持 mock 测试和多种实现切换。

**核心工厂函数：**

| 函数 | 说明 |
|------|------|
| `createSandboxInstance(config?)` | 创建完整沙盒实例（会话 + 工具提供者），主要工厂函数 |
| `createSandboxSession(config?)` | 创建沙盒会话实例，优先返回 mock 实例 |
| `createSandboxToolProvider(session?, config?)` | 创建工具提供者实例，优先返回 mock 实例 |

**Mock 支持函数：**

| 函数 | 说明 |
|------|------|
| `setMockSandboxSession(session)` | 注入 mock 会话实例（传 `null` 清除） |
| `setMockToolProvider(provider)` | 注入 mock 工具提供者实例（传 `null` 清除） |
| `clearAllMocks()` | 清除所有 mock 设置（测试后清理用） |

**内部适配器类：**

| 类 | 说明 |
|------|------|
| `DefaultSandboxSession` | 适配器，将现有 `SandboxSessionManager`（单例）包装为 `SandboxSessionInterface` |
| `DefaultToolProvider` | 适配器，将现有 `getSandboxToolsWithContext` + `isSandboxEnabled` 适配为 `SandboxToolProviderInterface` |

**工厂模式流程：**
```
createSandboxInstance(config?)
    ├── createSandboxSession(config?)
    │     ├── mockSession 已设置? → 返回 mock 实例
    │     └── 否则 → new DefaultSandboxSession()（包装 getSandboxManager()）
    └── createSandboxToolProvider(session, config?)
          ├── mockToolProvider 已设置? → 返回 mock 实例
          └── 否则 → new DefaultToolProvider()（包装 getSandboxToolsWithContext + isSandboxEnabled）
```

### 5.3 Skill Loader（skill-loader.ts）

**功能：** 在对话开始时将 Skill 文件加载到沙盒工作区。

**核心函数：**

| 函数 | 说明 |
|------|------|
| `loadSkillsToSandbox(userId, agentId, sessionId)` | 主入口，加载 Agent 配置的所有 Skill 到沙盒 |
| `getAgentSkillInfos(agentId)` | 获取 Agent 关联的 Skill 信息（含 fileHash） |
| `isSkillUpToDate(sessionId, userId, skillId, expectedHash)` | 检查沙盒中 Skill 是否已存在（版本检测） |
| `downloadSkillDirectoryToSandbox(...)` | 从 Supabase Storage 下载 Skill 目录并写入沙盒 |
| `generatePresetPrompt(skills)` | 生成预置提示词，告知 AI 已加载的 Skill |

**版本检测机制：**
- 使用 `test -d` 和 `test -f` 命令检查 `skills/{skillId}/SKILL.md` 是否存在
- 若目录和关键文件存在，跳过下载（避免重复加载）
- `fileHash` 字段预留用于未来精确版本检测

**预置提示词中的文件路径：**
- 统一使用 `skills/{skillId}/SKILL.md`（大写）
- 版本检测、预置提示词和实际文件名保持一致
- AI 可直接按预置提示词中的路径读取 Skill 文件

**加载流程：**
1. 获取 Agent 配置的 Skill 列表
2. 逐个检查是否已在沙盒中存在
3. 对不存在或过期的 Skill，从 Supabase Storage 下载
4. 创建目录结构并写入所有文件
5. 生成预置提示词供 AI 调用时使用

**返回结果：**
```typescript
interface LoadSkillsResult {
  success: boolean;           // 是否成功（部分失败也视为成功）
  loadedSkills: string[];     // 成功加载的 skillId 列表
  skippedSkills: string[];    // 跳过的 skillId 列表（已是最新）
  errors: Array<{ skillId: string; error: string }>;  // 加载失败的 Skill
  presetPrompt: string;       // 生成的预置提示词
}
```

当启用开关关闭或调用失败时，工具 `execute` 可返回 **友好字符串** 而非抛到前端崩溃（见 `tools.ts`）。

## 6. Skill 加载流程

### 6.1 整体流程图

```
Agent Chat 开始
    ↓
loadSkillsToSandbox(userId, agentId, sessionId)
    ↓
获取 Agent 关联的 Skill 信息
    ↓
遍历每个 Skill:
    ├── isSkillUpToDate() → true → 跳过（加入 skippedSkills）
    └── isSkillUpToDate() → false → 下载并写入沙盒
         ↓
    downloadSkillDirectory(userId, skillName)
         ↓
    创建目录: mkdir -p -m 777 skills/{skillId}
         ↓
    遍历文件: sandbox.writeFile(...)
         ↓
    加入 loadedSkills 或 errors
    ↓
generatePresetPrompt() → 生成预置提示词
    ↓
返回 LoadSkillsResult
```

### 6.2 Skill Validator（lib/infra/skills/validator.ts）

**功能：** Skill 文件格式校验工具，验证 Skill 文件结构是否符合规范。已迁移至 `lib/infra/skills/` 模块。

**核心函数：**

| 函数 | 说明 |
|------|------|
| `validateSkillFile(content)` | 校验单个 Skill 文件（Markdown + frontmatter） |
| `validateSkillDirectory(files)` | 校验 Skill 目录（必须有 SKILL.md） |
| `extractSkillMetadata(content)` | 从 Skill 内容提取元数据 |
| `extractSkillBody(content)` | 获取 Skill 正文（不含 frontmatter） |

**校验约束：**

| 约束 | 值 | 说明 |
|------|------|------|
| NAME_MIN_LENGTH | 1 | Skill 名称最小长度 |
| NAME_MAX_LENGTH | 50 | Skill 名称最大长度 |
| DESCRIPTION_MIN_LENGTH | 1 | 描述最小长度 |
| DESCRIPTION_MAX_LENGTH | 200 | 描述最大长度 |
| MAX_FILE_SIZE | 5MB | 单文件大小限制 |
| MAX_TOTAL_SIZE | 5MB | 目录总大小限制 |

**frontmatter 必填字段：**
- `name`: Skill 名称（1-50 字符）
- `description`: Skill 描述（1-200 字符）

**功能：** 提供完整的 Skill 能力，包括类型定义、加载执行、发现验证、工具定义和服务层。

| 文件 | 职责 |

3. 为每个 Skill 创建专属工具（`skill_{skillId}`）并注册
4. 通过 `createTools` 从 ToolRegistry 创建工具实例

## 7. 与 Agent Chat 集成

**入口文件：** `app/api/agent-chat/route.ts`
**运行时文件：** `lib/agent-chat/runtime.ts`

**集成流程：**

1. `route.ts` 解析请求、认证、加载 Agent 配置、确保对话存在。
2. 调用 `createRuntime()` 创建运行时上下文：
   - 加载 Skills 到沙盒（`loadSkillsToSandbox`，如果沙盒启用且 Agent 有配置 Skill）
   - 构建系统提示词（含 Skill 预置提示词）
   - 直接调用 `getSandboxToolsWithContext({ conversationId, userId })` 创建沙盒工具
   - 合并 MCP 运行时工具（best-effort，失败时降级为仅系统工具）
3. `executeAgent()` 使用 `ToolLoopAgent` 执行流式响应。

**关键代码（`lib/agent-chat/runtime.ts`）：**
```typescript
import { getSandboxToolsWithContext, loadSkillsToSandbox, isSandboxEnabled } from "@/lib/infra/sandbox";

// 创建沙盒工具（绑定会话上下文）
const sandboxTools = getSandboxToolsWithContext({ conversationId, userId });
```

因此：**沙盒工具始终绑定当前对话 id 与用户 id**；未创建对话前需先有 `conversationId`（该路由要求 body 带对话 id，与客户端约定一致）。

**注意：** 旧的工具入口封装层 `lib/agents/tools/sandbox.ts` 已删除。Agent Chat 现在直接从 `lib/infra/sandbox` 导入沙盒工具函数，不再需要中间适配层。

## 8. 安全与运维

### 8.1 基本安全措施

1. **API Key** 仅走服务端环境变量，勿下发浏览器。
2. **nsjail** 限制 CPU/内存/进程与网络（注意 README 中 **`clone_newnet: false`** 时与宿主机共享网络——生产请按需收紧）。
3. **HTTPS：** 公网 Gateway 建议使用 TLS。
4. **日志：** Next 与 Python 服务分别排查；先看 Gateway 4xx/5xx 与 Next 控制台 `沙盒服务未启用` 类返回。

### 8.2 路径验证机制（lib/infra/sandbox/path-validator.ts）

**安全验证模块，防止路径遍历攻击和命令注入攻击。** 原路径 `lib/utils/sandbox-path-validator.ts` 已迁移至 `lib/infra/sandbox/path-validator.ts`。

**验证规则：**

| 规则 | 说明 |
|------|------|
| 路径长度限制 | 最大 256 字符 |
| 禁止字符检测 | 检测 `;`、`|`、`$`、`\`、`'`、`"`、`(`、`)`、`&`、`<`、`>`、`` ` ``、`\n`、`\r` 等（命令注入防护） |
| 非法序列检测 | 检测 `..`、`~`（路径遍历防护） |
| 绝对路径检查 | 路径必须以 `/` 开头 |
| 沙盒范围检查 | 规范化后路径必须在 `/workspace` 范围内 |

**核心函数：**

| 函数 | 说明 |
|------|------|
| `validateSandboxPath(userPath)` | 验证沙盒路径安全性，返回 `{ valid, error? }` |
| `hashUserId(userId)` | 对用户 ID 进行 SHA256 哈希（返回前 16 位） |

**使用示例：**
```typescript
import { validateSandboxPath, hashUserId } from '@/lib/infra/sandbox';

// 验证路径
const result = validateSandboxPath('/skills/test/skill.md');
if (!result.valid) {
  console.error(result.error);
}

// 哈希用户 ID（用于目录命名）
const hashedUserId = hashUserId('user-123');
```

## 9. 故障排查

| 现象 | 检查 |
|------|------|
| 工具返回「沙盒服务未启用」 | `SANDBOX_ENABLED` 是否为字符串 `true`，URL/API Key 是否通过 `validateSandboxConfig` |
| HTTP 超时 | `SANDBOX_REQUEST_TIMEOUT_MS`、网关负载与 nsjail `time_limit` |
| exec 无输出 / exit 非 0 | 看 Gateway 返回的 stdout/stderr 拼装逻辑 |
| 权限 / nsjail 启动失败 | `sandbox-service/README.md` 目录权限与二进制兼容性 |
| Skill 加载失败 | 检查 Supabase Storage 路径、Skill 目录是否包含 SKILL.md |
| 路径验证失败 | 检查路径是否包含禁止字符或非法序列 |
