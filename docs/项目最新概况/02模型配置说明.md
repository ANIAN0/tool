# 用户模型配置说明

> **文档目的：** 说明用户自定义 LLM 配置如何入库、加密、在 Agent 对话中选用。  
> **系统概述：** 每个用户可在设置页维护多条 `user_models`；其中一条可设为默认。Agent 可绑定创建者的一条模型；未绑定时 Agent Chat 使用当前用户默认模型；若无默认模型则直接报错提示配置。模型创建存在两条路径：Agent Chat 通过 `lib/agent-chat/model-resolver.ts` 中的 `buildChatModelFromUserModel` 直接构建；V1 Chat 通过 `ModelService.createModel()` 统一入口构建。当前仅支持 OpenAI-Compatible 协议。  
> **技术栈：** Next.js、Turso(libSQL)、`@ai-sdk/openai-compatible`、AES-256-GCM 加密（`lib/encryption.ts`）、ModelService（`lib/infra/model/`）

## 1. 系统架构

| 模块 | 路径 | 职责 |
|------|------|------|
| Schema | `lib/schemas/model.ts` | `user_models` 表 DDL 与 `UserModel` 等类型定义（`lib/db/schema.ts` 为兼容层，统一转发至 `lib/schemas/`） |
| ModelService | `lib/infra/model/index.ts` | **V1 Chat 模型创建入口**，管理 Provider 注册与模型实例创建，提供 `createModel` 便捷函数，聚合导出所有类型 |
| Provider 注册表 | `lib/infra/model/provider-registry.ts` | Provider 注册/查找/注销，管理 `ProviderType` 类型定义（当前运行时仅使用 `openai`） |
| 模型解析器 | `lib/infra/model/resolver.ts` | 根据模型 ID 查找 `ModelConfig` → 获取对应 Provider → 创建 `LanguageModel` 实例；支持 `resolveUserModel` 从数据库直接解析 |
| 模型中间件 | `lib/infra/model/middleware.ts` | LLM 调用中间件：DevTools 日志、压缩检测（60% 阈值触发） |
| Token 估算 | `lib/infra/model/token-estimation.ts` | 基于固定系数的 Token 数估算（ASCII 4:1，非 ASCII 1.5:1） |
| 用户 Provider | `lib/infra/model/user-provider.ts` | 根据用户模型配置创建 OpenAI-Compatible Provider，包含配置验证 |
| Agent Chat 模型解析 | `lib/agent-chat/model-resolver.ts` | **Agent Chat 模型创建入口**，通过 `buildChatModelFromUserModel` 直接构建模型（不经过 ModelService），提供 `resolveModel` 解析 Agent 绑定或用户默认模型 |
| CRUD | `lib/db/user-models.ts` | 增删改查、设置默认模型（使用 `React.cache()` 和 `unstable_cache()` 缓存优化） |
| 加密 | `lib/encryption.ts` | AES-256-GCM 加密 + scrypt 密钥派生，API Key 写入前加密、读出解密 |
| Agent Chat 选型 | `app/api/agent-chat/route.ts` | 通过 `lib/agent-chat/model-resolver.ts` 的 `resolveModel` 构建模型，支持压缩检测中间件 |
| V1 Chat 选型 | `app/api/v1/chat/route.ts` | 通过 `ModelService.createModel()` 构建模型，采用 API Key 鉴权（对外接口） |
| 设置 UI | `app/settings/models/**` | 列表与表单（Provider 固定为 OpenAI-Compatible） |

**流程简述：** 用户在设置页保存模型（Provider 固定为 openai）→ 密钥加密入库 → 运行时存在两条模型构建路径：
- **Agent Chat**：`resolveModel()` → `buildChatModelFromUserModel()` → 解密 API Key → `createOpenAICompatible()` 创建 Provider → 生成 `LanguageModel` → 包装中间件（压缩检测）→ 流式对话
- **V1 Chat**：`ModelService.createModel()` → `createUserModelProvider()` 创建 Provider → 生成 `LanguageModel` → 包装 DevTools 中间件 → 流式对话

### 1.1 ModelService 架构

```
ModelService (单例)
  ├── providers: Map<string, ProviderInfo>
  │     ├── id: string
  │     ├── type: ProviderType
  │     └── createModel: (modelId) => LanguageModel
  ├── registerProvider(id, registration) → 注册 Provider
  └── createModel({modelId, userId, wrapDevTools?}) → 创建模型实例
        ├── 1. getUserModels(userId) → 校验模型归属
        ├── 2. createUserModelProvider(userModel) → 创建 Provider
        ├── 3. provider(userModel.model) → 生成 LanguageModel
        ├── 4. wrapModelWithDevTools(model) → 开发环境自动包装
        └── 返回 { model, modelName, contextLimit, cleanup }
```

### 1.2 中间件架构

```
wrapModelWithAllMiddlewares(model, options?)
  ├── createCompressionDetectionMiddleware({conversationId, contextLimit})
  │     ├── estimateTotalTokensFromParams(params) → Token 估算
  │     ├── 超过 60% 阈值 → createCompressionTask()
  │     └── 继续执行 doStream()
  ├── logInputMiddleware (开发环境)
  │     ├── wrapGenerate → 打印消息/工具数量
  │     └── wrapStream → 打印消息/工具数量
  └── devToolsMiddleware() (开发环境)
```

### 1.3 Agent Chat 模块

**路径：** `lib/agent-chat/`

Agent Chat 的独立模块，提供请求解析、认证验证、配置加载、模型解析、会话管理、运行时创建、响应构建等完整功能。模型构建**不经过 ModelService**，而是使用独立的 `buildChatModelFromUserModel` 函数。

| 文件 | 职责 |
|------|------|
| `model-resolver.ts` | 模型解析与构建：`resolveModel` 根据 Agent 绑定或用户默认模型解析配置，`buildChatModelFromUserModel` 直接创建 OpenAI-Compatible 模型实例 |
| `agent-loader.ts` | Agent 配置加载 |
| `auth-context.ts` | 认证上下文获取 |
| `conversation.ts` | 会话管理（创建、加载历史、保存消息） |
| `runtime.ts` | 运行时创建与 Agent 执行 |
| `request.ts` | 请求体解析 |
| `response.ts` | 流式响应构建 |
| `types.ts` | 类型定义 |
| `utils.ts` | 工具函数 |
| `index.ts` | 统一导出入口 |

**与 V1 Chat 的模型构建差异：**

| 对比项 | Agent Chat | V1 Chat |
|--------|-----------|---------|
| 入口函数 | `buildChatModelFromUserModel()` | `ModelService.createModel()` |
| 所在模块 | `lib/agent-chat/model-resolver.ts` | `lib/infra/model/index.ts` |
| Provider 创建 | 直接调用 `createOpenAICompatible()` | 通过 `createUserModelProvider()` |
| 模型归属校验 | `resolveModel` 中按 `agentOwnerId` 查询 | `createModel` 中按 `userId` 查询 |
| DevTools 包装 | 不自动包装 | 开发环境自动包装 |

### 1.4 API 认证方式对比

| API | 路径 | 认证方式 | 适用场景 |
|------|------|------|------|
| Agent Chat | `/api/agent-chat` | 可选鉴权中间件（JWT+匿名） | 内部 Web 应用调用 |
| V1 Chat | `/api/v1/chat` | API Key 鉴权（`authenticateApiKey`） | 对外开放接口，第三方集成 |

**说明：** 两种 API 使用不同的模型构建路径，模型选择逻辑存在关键差异（详见 1.3 节对比表和 3.4 节）。V1 API 通过 `Authorization: Bearer sk_live_xxx` 请求头进行身份验证，适用于需要对外开放的集成场景。

## 2. 数据库设计

表 `user_models`（节选，完整见 `lib/schemas/model.ts`）：

| 字段 | 说明 |
|------|------|
| `id` | 主键 |
| `user_id` | 所属用户 |
| `name` | 展示名称 |
| `provider` | 提供商类型字符串；当前仅支持 `openai`（代表 OpenAI-Compatible 协议） |
| `model` | 模型名（如 `gpt-4o-mini`、`deepseek-chat`、`claude-3-5-sonnet` 等，任何 OpenAI 兼容模型均可） |
| `api_key` | **加密**后的密钥 |
| `base_url` | 可选；OpenAI-Compatible 端点地址，留空默认使用 `https://api.openai.com/v1` |
| `is_default` | 是否默认模型 |
| `context_limit` | 模型上下文上限（token数）；数据库类型 `INTEGER`，默认值 32000，TypeScript 类型 `number`。用于会话压缩检测，当累计 token 达到 60% 阈值时触发压缩任务 |

### 2.1 TypeScript 类型定义

```typescript

  is_default: boolean;
```

## 3. 后端实现

### 3.0 加密算法说明

**文件：** `lib/encryption.ts`

API Key 采用 **AES-256-GCM** 认证加密算法存储，确保密钥安全：

| 参数 | 值 | 说明 |
|------|-----|------|
| 算法 | AES-256-GCM | 认证加密模式，防篡改 |
| 密钥派生 | scrypt | 从主密钥 + 盐值派生 32 字节密钥 |
| IV 长度 | 16 字节（128位） | 初始化向量 |
| 盐值长度 | 32 字节（256位） | 每次加密生成随机盐 |
| 认证标签 | 16 字节（128位） | GCM 模式认证标签 |

**加密流程：**
1. 从环境变量 `ENCRYPTION_KEY` 获取主密钥（64 字符 hex）
2. 生成随机盐值（32 字节）
3. 使用 scrypt 派生实际加密密钥
4. 生成随机 IV（16 字节）
5. AES-256-GCM 加密，获取密文 + 认证标签
6. 组合：盐值 + IV + 认证标签 + 密文 → Base64 编码存储

**安全说明：**
- 主密钥必须为 32 字节（64 字符 hex）
- 每次加密使用不同盐值和 IV，相同明文产生不同密文
- GCM 认证标签确保密文未被篡改

### 3.1 ModelService 与 Provider 构建

**核心文件：** `lib/infra/model/`

V1 Chat 的模型构建通过 **ModelService + 用户 Provider** 调度（Agent Chat 使用独立的 `lib/agent-chat/model-resolver.ts`，详见 1.3 节）。当前仅支持 OpenAI-Compatible 协议：

1. **ModelService**（`index.ts`）
   - `ModelService` 类为单例，提供统一的模型创建入口
   - `createModel({modelId, userId, wrapDevTools?})` 完成校验 → Provider 创建 → 模型生成 → 中间件包装
   - 校验逻辑：先查用户模型列表确认归属，失败则抛出 `ModelServiceError`
   - 开发环境自动包装 DevTools 中间件（可通过 `wrapDevTools` 参数覆盖）
   - 返回 `{ model, modelName, contextLimit, cleanup }`

2. **用户 Provider**（`user-provider.ts`）
   - `SupportedProvider` 类型仅包含 `"openai"`
   - `createUserModelProvider(userModel)` 解密 API Key → 确定 baseURL → 调用 `createOpenAICompatible()` 创建 Provider
   - 未配置 `base_url` 时默认使用 `https://api.openai.com/v1`
   - `validateUserModel(userModel)` 验证配置有效性

3. **Provider 注册表**（`provider-registry.ts`）
   - `ProviderType` 类型定义保留六种：`openai | anthropic | google | azure | vercel | custom`
   - **实际运行时仅使用 `openai`**，其他类型仅作类型定义保留
   - `providerRegistry` 为单例 Map，通过 `register` / `get` / `has` 操作

4. **模型解析器**（`resolver.ts`）
   - `resolveModel(modelId, userId?)` 从内存缓存查找 `ModelConfig` → 获取 Provider → 创建模型
   - `resolveUserModel(modelId, userId)` 直接从数据库查询用户模型 → 调用 `createUserModelProvider` → 创建模型
   - 支持默认 Provider 兜底

### 3.2 模型中间件

**文件：** `lib/infra/model/middleware.ts`

提供 LLM 调用的中间件逻辑，支持 DevTools 日志和压缩检测：

1. **压缩检测中间件**（`createCompressionDetectionMiddleware`）
   - 工厂函数，接收 `{conversationId, contextLimit}` 参数
   - 在流式调用前通过 `estimateTotalTokensFromParams` 估算 token 总数
   - 超过上下文上限 60% 阈值时，调用 `createCompressionTask()` 创建压缩任务
   - 唯一约束冲突（已有未处理任务）时静默忽略

2. **日志中间件**（`logInputMiddleware`）
   - 仅在开发环境启用
   - `wrapGenerate` / `wrapStream` 钩子打印消息数量和工具数量

3. **DevTools 中间件**（`devToolsMiddleware`）
   - 来自 `@ai-sdk/devtools`，仅在开发环境启用
   - 用于调试 LLM 请求、响应及工具调用过程

4. **组合包装函数**
   - `wrapModelWithAllMiddlewares(model, options?)` — 包装所有中间件（压缩检测 + DevTools）
   - `wrapModelWithDevTools(model)` — 仅包装 DevTools 中间件

### 3.3 Token 估算

**文件：** `lib/infra/model/token-estimation.ts`

基于固定系数的 Token 数估算，用于压缩检测中间件：

| 字符类型 | 估算规则 |
|---------|---------|
| ASCII 字符 | 4 字符 ≈ 1 token |
| 非 ASCII 字符（中文等） | 1.5 字符 ≈ 1 token |

**估算范围**（由 `estimateTotalTokensFromParams` 处理）：
- Prompt（messages）：文本内容、图片（约 85 tokens）、文件（约 50 tokens）、tool-call/tool-result
- Tools：工具名称、描述、inputSchema 或 args

### 3.4 默认与失败处理

**Agent Chat**（`lib/agent-chat/model-resolver.ts` 的 `resolveModel`）：
1. Agent 配置了 `model_id`：按 `agentOwnerId` 查询该模型 → 不存在则返回 400 错误。
2. Agent **未**配置模型：取当前用户的 **`getDefaultUserModel(userId)`**；若存在且可构建则用之。
3. 无默认或构建失败（如 provider 不支持）：直接返回 400，提示用户先在设置页配置并设为默认模型。

**V1 Chat**（`ModelService.createModel()`）：
1. 按 `userId` + `modelId` 查询模型 → 不存在或不属于该用户则抛出 `ModelServiceError`。
2. 无可用模型时返回 400 错误。

### 3.5 V1 Chat API（对外接口）

**文件：** `app/api/v1/chat/route.ts`

- **认证方式**：使用 `authenticateApiKey` 中间件进行 API Key 鉴权，请求头格式为 `Authorization: Bearer sk_live_xxx`。
- **速率限制**：每分钟最多 10 次请求，连续失败 5 次后锁定 15 分钟。
- **模型选择逻辑**：与 Agent Chat **存在关键差异**：
  1. 若 Agent 绑定 `model_id`，从 **API Key 用户** 的模型池获取：`getUserModelById(userId, agent.model_id)`
  2. 若无绑定模型，取 **API Key 用户** 的默认模型：`getDefaultUserModel(userId)`
  3. 无可用模型时返回 400 错误
- **差异对比**：
  - Agent Chat：绑定模型时使用 `agent.user_id`（Agent 创建者的模型池）
  - V1 Chat：绑定模型时使用 `userId`（API Key 用户的模型池）
- **适用场景**：对外开放的对话接口，支持第三方系统集成，通过 API Key 进行身份认证。

#### 3.5.1 V1 API 错误码

| 错误码 | HTTP 状态 | 说明 |
|--------|----------|------|
| `UNAUTHORIZED` | 401 | 未提供 API Key |
| `INVALID_API_KEY` | 401 | API Key 格式无效或已过期 |
| `RATE_LIMITED` | 429 | 请求过于频繁，已被锁定（当前实现统一返回 401，待修正为 429） |
| `RATE_LIMIT_EXCEEDED` | 429 | 请求次数超限（当前实现统一返回 401，待修正为 429） |

### 3.6 用户模型 HTTP API

路径前缀 **`/api/user/models`**（含列表、创建、更新、删除、设默认、同步等子资源），当前均使用可选鉴权中间件，但业务上依赖用户身份（匿名/登录均会落到具体 `userId`）。

### 3.7 用户模型 CRUD 缓存优化

**文件：** `lib/db/user-models.ts`

| 函数 | 缓存策略 | 说明 |
|------|---------|------|
| `getUserModelById` | `React.cache()` | 同一请求内缓存，避免重复查询 |

## 5. 使用指南

1. 在 **设置 → 模型** 添加一条模型配置（Provider 固定为 OpenAI-Compatible）。
2. 配置参数说明：
   - **名称**：模型的展示名称（便于识别）
   - **Provider**：固定为 OpenAI-Compatible（`openai`），所有兼容 OpenAI 协议的服务均可使用
   - **API Key**：提供商的密钥（加密存储）
   - **Base URL**：可选，用于指定自定义端点（如 DeepSeek、本地 Ollama 等 OpenAI 兼容端点），留空默认使用 `https://api.openai.com/v1`
   - **模型名**：具体模型标识（如 `gpt-4o-mini`、`deepseek-chat`、`claude-3-5-sonnet` 等）
   - **上下文上限**：可选设置，创建时未传入则使用默认值 32000；读取后始终返回数值。用于会话压缩检测（当累计 token 达到上限的 60% 时触发压缩任务），避免超出模型上下文窗口限制
3. 可选：设为默认，供未指定模型的 Agent 使用。
4. 在 **Agent 配置** 中为 Agent 选择关联模型（见 `06agent配置说明.md`）。

## 6. 注意事项

1. **密钥仅存储密文**；日志中勿打印解密后的 key。  
2. **OpenAI-Compatible 不等于仅限 OpenAI 官方**：任意 OpenAI 兼容端点（如 DeepSeek、Ollama、vLLM 等）均可通过 `base_url` 指向，使用 `openai` Provider 类型。  
3. 当前不再提供 `OPENROUTER_API_KEY` 的对话降级路径；必须有可用用户模型。
4. **Provider 类型已简化**：当前运行时仅支持 `openai`（OpenAI-Compatible），`ProviderType` 类型定义中的其他类型（anthropic、google、azure、vercel、custom）仅作保留，实际不可用。前端表单已移除 Provider 选择器。
5. **旧 LocalStorage 方案已移除**：`lib/settings/user-models-storage.ts` 已删除，所有模型配置统一走数据库 + 加密存储，不再有浏览器本地存储的降级路径。
6. **Schema 来源变更**：类型定义和 DDL 统一由 `lib/schemas/model.ts` 提供，`lib/db/schema.ts` 现为兼容层（仅转发导出），新代码应直接从 `@/lib/schemas` 导入。
7. **模块路径迁移**：模型核心模块已从 `lib/models/` 迁移至 `lib/infra/model/`，旧路径不再可用。
