# 会话压缩与消息管理说明

> 本文档记录会话压缩持久化、消息撤回、Token统计等功能的完整实现，为开发者提供技术参考和运维指南。

---

## 系统概述

本项目实现了完整的会话生命周期管理，包括：
- **lib/infra 服务抽象层**：在 lib/db 数据访问层之上，提供 MessageService、SessionService 等统一接口，解耦业务逻辑与数据层
- **会话压缩持久化**：异步任务调度机制，通过 compression_tasks 和 checkpoints 表管理压缩状态
- **消息撤回归档**：级联删除机制，被撤回消息归档到 deleted_messages 表
- **Token统计估算**：固定系数估算算法，支持实时检测和汇总统计
- **上下文上限配置**：基于 user_models.context_limit 字段的阈值控制

### 分层架构说明

项目采用分层架构，从上到下依次为：

1. **业务层（app/）**：API 路由和前端页面，调用 lib/infra 服务
2. **服务抽象层（lib/infra/）**：统一的服务接口，封装业务逻辑，解耦上层与数据层
3. **数据访问层（lib/db/）**：直接操作数据库的 CRUD 函数
4. **数据库（Turso/LibSQL）**：数据持久化存储

这种分层设计使得：
- 业务层不直接依赖数据库操作，便于测试和维护
- 不同类型的会话（Chat、Workflow）可以有各自的服务实现
- 数据层变更不会影响上层业务逻辑

### 压缩检测流程

1. **LLM调用前**：压缩检测中间件估算当前 token 数量
2. **阈值判断**：若超过 60% 上下文上限，创建压缩任务
3. **任务调度**：任务进入 pending 状态，等待执行
4. **压缩执行**：移除 50% 最早消息，创建 Checkpoint
5. **缓存更新**：更新 compression_cache 字段

### 消息撤回流程

1. **API调用**：DELETE /api/messages/[id]/delete
2. **权限验证**：验证用户是否拥有对话所有权
3. **级联删除**：删除目标消息及之后所有消息
4. **归档存储**：被删除消息写入 deleted_messages 表

## lib/infra 服务抽象层

lib/infra 层在 lib/db 数据访问层之上提供统一的服务接口，将业务逻辑与数据操作分离。包含两个核心模块：`lib/infra/message`（消息服务）和 `lib/infra/session`（会话服务）。

### MessageService（消息服务）

消息服务封装消息的 CRUD 和撤回操作，提供 UIMessage 格式转换能力。

#### 接口定义（lib/infra/message/types.ts）

```typescript
/**
 * 消息服务接口
 */
export interface MessageService {
  /** 创建新消息 */
  create(params: CreateMessageParamsInput): Promise<Message>;

  /** 获取会话的所有消息（按创建时间升序） */
  listBySession(sessionId: string): Promise<Message[]>;

  /** 删除会话的所有消息，返回删除数量 */
  deleteBySession(sessionId: string): Promise<number>;

  /** 将数据库消息转换为 UIMessage 格式 */
  toUIMessage(msg: Message, metadata?: { contextLimit: number; modelName: string }): UIMessage;

  /** 撤回消息（级联删除） */
  retract(messageId: string, userId: string, reason: "user-delete" | "edit-regenerate"): Promise<boolean>;
}
```

#### 关键类型


#### 实现说明（lib/infra/message/service.ts）

`MessageServiceImpl` 类实现 `MessageService` 接口，内部委托给 `lib/db/messages` 和 `lib/db/message-retract` 模块：

- `create()` 委托给 `dbMessages.createMessage()`
- `listBySession()` 委托给 `dbMessages.getMessages()`
- `deleteBySession()` 委托给 `dbMessages.deleteMessagesByConversation()`
- `retract()` 委托给 `dbRetract.retractMessage()`，返回 `deletedCount > 0`
- `toUIMessage()` 处理三种消息格式：checkpoint 类型、JSON 格式（UIMessage）、纯文本格式（兼容旧数据）


### compression_tasks 表

存储压缩任务状态，用于异步任务调度。

```sql
CREATE TABLE IF NOT EXISTS compression_tasks (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  status INTEGER DEFAULT 0,         -- 0: pending, 1: completed
  created_at INTEGER NOT NULL,
  completed_at INTEGER,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

-- 唯一索引：每个会话只能有1个未处理的压缩任务
CREATE UNIQUE INDEX IF NOT EXISTS idx_compression_tasks_pending_unique
ON compression_tasks(conversation_id, status)
WHERE status = 0;
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT | 任务ID（nanoid生成） |
| conversation_id | TEXT | 关联会话ID |
| status | INTEGER | 0=pending, 1=completed |
| created_at | INTEGER | 创建时间戳 |
| completed_at | INTEGER | 完成时间戳 |

### checkpoints 表

存储压缩检查点记录。cache_content 字段通过后续迁移添加，便于排查问题。

```sql

CREATE INDEX IF NOT EXISTS idx_checkpoints_conversation
ON checkpoints(conversation_id, created_at DESC);
```

| 字段 | 类型 | 说明 |

1. 加载历史消息（cache + checkpoint后的消息）
2. 计算移除数量（50%，至少保留1条）
3. 创建 Checkpoint 记录（含 cache_content）
4. 更新 compression_cache 字段

```typescript
export function createCompressionDetectionMiddleware(options: {
  conversationId: string;
  contextLimit: number;
}): LanguageModelV3Middleware {
  const triggerThreshold = Math.floor(contextLimit * 0.6); // 60%阈值

  return {
    specificationVersion: 'v3',
    wrapStream: async ({ doStream, params }) => {
      const totalTokens = estimateTotalTokensFromParams(params);
      if (totalTokens > triggerThreshold) {
        await createCompressionTask({ id: nanoid(), conversationId });
      }
      return doStream();
    },
  };
}
```

#### getConversationsWithFilter 逻辑

该函数用于获取用户对话列表并支持 source 过滤，使用 `React.cache()` 缓存查询结果：

1. 指定 source 时：按 `user_id` + `source` 过滤
2. 未指定 source 时：兼容旧数据，过滤 `source IS NULL OR source = 'chat'`
3. 结果按 `updated_at DESC` 排序

```typescript
export const getConversationsWithFilter = cache(async (
  userId: string,
  options?: { source?: string }
): Promise<Conversation[]> => {
  let sql = `SELECT * FROM conversations WHERE user_id = ?`;
  const args: string[] = [userId];

  if (options?.source) {
    sql += ` AND source = ?`;
    args.push(options.source);
  } else {
    // 未指定source时，兼容旧数据
    sql += ` AND (source IS NULL OR source = 'chat')`;
  }

  sql += ` ORDER BY updated_at DESC`;

  const result = await db.execute({ sql, args });
  return result.rows.map(rowToConversation);
});
```

#### CreateConversationParams 类型说明

agentId 在 schema 层和服务层有不同的类型要求：
- **schema 层（lib/schemas/conversation.ts）**：`CreateConversationParams.agentId` 为**必填字段**，对应数据库 `agent_id TEXT NOT NULL` 约束
- **服务层（lib/infra/session/types.ts）**：`CreateSessionParams.agentId` 为**可选字段**（`agentId?: string`），允许创建会话时暂不指定 agent
- **数据库层（lib/db/conversations.ts）**：`createConversation` 函数要求 `agentId` 必填（schema层约束）

```typescript
// schema 层：必填（对应数据库 NOT NULL 约束）
```

## 注意事项

1. **幂等性处理**：
   - 消息已不存在时，撤回API返回 deletedCount=0
   - 唯一约束冲突时，压缩任务创建被忽略

2. **事务原子性**：
   - 消息撤回使用 batch 执行，保证归档和删除的原子性
   - 压缩执行涉及 Checkpoint 和 cache 更新

3. **压缩不影响撤回**：
   - 用户只能删除 checkpoint 之后的消息
   - cache 存储的是 checkpoint 之前的消息
   - 删除操作不影响缓存内容

4. **Token估算精度**：
   - 固定系数估算，实际值可能有偏差
   - 主要用于阈值检测，非精确统计

5. **上下文上限配置**：
   - 通过 user_models.context_limit 字段设置
   - 默认值 32000 tokens
   - 压缩触发阈值为 60%（即 19200）

6. **agentId 类型差异**：
   - schema 层 `CreateConversationParams.agentId` 为**必填**（数据库 `agent_id TEXT NOT NULL` 约束）
   - 服务层 `CreateSessionParams.agentId` 为**可选**（`agentId?: string`，允许暂不指定 agent）
   - 数据库层 `rowToConversation` 直接使用数据库值（Session 类型中 agentId 为必填，因为数据库已有值）
   - 前端调用数据库层时必须传入 agentId；通过服务层创建会话时 agentId 可选

7. **服务层分层**：
   - 业务层应通过 `lib/infra` 服务访问数据，不直接调用 `lib/db`
   - 不同会话类型使用对应的服务实现：ChatSessionService / WorkflowSessionService
